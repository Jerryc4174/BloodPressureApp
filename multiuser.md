## Adding multi-user capabiltiy to app.

### Proposed Approach

The existing application is a single user app.  I would like to 
enhance it to allow two named users to be able to use it.

To accomplish this, I plan to add a field named: User to the database
an make it a second key in the database which current uses Date as
the primary key.

I would like the program to be modified to have a start up dialog with
two buttons.  Button1 will identify user Jerry and have that label.  The second button will identify user Wanda and have that label. 

After the user presses one of these buttons, the program will go to the existing main page and use 
that user name to search for entries in the database selecting only
those entries for all CRUD operations.  To maintain database compatibility user Jerry will enter Jerry for all new entries but will select the records for CRUD for all entries with either or NULL(no entry).

### Expected Database Schema (for code implementation)

The code now expects table `public.data` to include these columns:

- `EntryId BIGINT` (unique row identifier)
- `UserId TEXT NOT NULL` (values currently expected: `Jerry` or `Wanda`)
- `Date TIMESTAMP` (existing reading datetime)
- `Upper INTEGER`
- `Lower INTEGER`
- `BPM INTEGER`

The code also expects:

- Unique index on `EntryId`
- Unique index on (`UserId`, `Date`) to prevent duplicate timestamps per user

Suggested Postgres migration SQL:

```sql
BEGIN;

-- 1) Add EntryId if missing, backfill, and enforce uniqueness.
ALTER TABLE public.data
ADD COLUMN IF NOT EXISTS "EntryId" BIGINT GENERATED BY DEFAULT AS IDENTITY;

UPDATE public.data
SET "EntryId" = DEFAULT
WHERE "EntryId" IS NULL;

CREATE UNIQUE INDEX IF NOT EXISTS data_entryid_uidx
ON public.data ("EntryId");

-- 1b) Move the primary key to EntryId.
-- If Date is currently the PK, drop that PK constraint first, then add PK on EntryId.
DO $$
DECLARE
	pk_name text;
BEGIN
	SELECT tc.constraint_name INTO pk_name
	FROM information_schema.table_constraints tc
	WHERE tc.table_schema = 'public'
	  AND tc.table_name = 'data'
	  AND tc.constraint_type = 'PRIMARY KEY'
	LIMIT 1;

	IF pk_name IS NOT NULL THEN
		EXECUTE format('ALTER TABLE public.data DROP CONSTRAINT %I', pk_name);
	END IF;

	BEGIN
		ALTER TABLE public.data ADD PRIMARY KEY ("EntryId");
	EXCEPTION
		WHEN duplicate_table THEN NULL;
		WHEN duplicate_object THEN NULL;
	END;
END $$;

-- 2) Add UserId if missing, backfill legacy rows to Jerry, and enforce NOT NULL.
ALTER TABLE public.data
ADD COLUMN IF NOT EXISTS "UserId" TEXT;

UPDATE public.data
SET "UserId" = 'Jerry'
WHERE "UserId" IS NULL;

ALTER TABLE public.data
ALTER COLUMN "UserId" SET NOT NULL;

-- 3) Prevent duplicate datetime rows for the same user.
CREATE UNIQUE INDEX IF NOT EXISTS data_userid_date_uidx
ON public.data ("UserId", "Date");

COMMIT;
```

Note:
- Keeping your original datetime value is fine; it no longer needs to be the table primary key.
- Moving to `EntryId` as row identifier will not cause functional problems for this app and simplifies user-scoped updates/deletes.
